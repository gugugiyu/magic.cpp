# Context Condenser — Concrete Implementation Plan (v5)

> Derived from v4 spec + UI spec (3.1), reviewed against the live codebase.
> Structured into 5 independent layers that can be assigned to an agent sequentially.

---

## Pre-Implementation: Findings & Resolved Ambiguities

Before assigning any layer, the following issues were found by comparing the v4 spec against the codebase. All are resolved here so the implementing agent has no ambiguity.

---

### Critical Fixes Applied to the Spec

#### 1. Pako compression — use WebWorker for async execution

**v4 spec says:** `raw: pako_encoded_conversation`

**Reality:** `pako` IS installed. However, compressing/decompressing large conversation snapshots on the main thread blocks the UI during active chat.

**Resolution:** Use `pako` (gzip deflate/inflate) for snapshot compression, but always run it inside a **Web Worker** (`src/lib/workers/condenser-compression.worker.ts`). The main thread posts a `{ type: 'compress' | 'decompress', payload }` message and awaits the result. For the Topic Map's `raw` field, store plain JSON string — the 10-entry cap keeps it small enough. The `pako` reference in the spec is kept; the WebWorker wrapper is the new addition.

---

#### 2. Layer 0 vs existing `subagentSummarizeEnabled`

**v4 spec says:** Layer 0 handles "large dump" inputs.

**Reality:** `subagent.service.ts` already implements `summarizeText()`, and `subagentSummarizeEnabled` already gates pre-send text summarization. The `shouldSummarize()` / `estimateTokens()` functions exist in `src/lib/utils/tokenizer.ts`.

**Resolution:** Layer 0 is an **upgrade** of the existing summarization path, not a parallel one. When the condenser is enabled and the subagent is configured, Layer 0 replaces the standalone `subagentSummarizeEnabled` check with a context-window-relative threshold. The condenser toggle effectively supersedes `subagentSummarizeEnabled` when active.

---

#### 3. Both compression model and main model are dynamic (configurable)

**v4 spec says:** Subagent performs all compression tasks.

**Reality:** Subagent endpoint is optional. The main model endpoint is also configurable (external API mode, local llama.cpp, etc.).

**Resolution:** Both endpoints are **dynamic**:

- When subagent **is** configured (`subagentConfigStore.isConfigured === true`): use the subagent endpoint for all compression tasks (preferred, lower latency).
- When subagent **is not** configured: fall back to the **main model endpoint** (same server used for chat) for compression. This makes the condenser functional without a separate subagent.

The settings UI shows the active compression endpoint:

- With subagent: "Using subagent model for compression"
- Without: "⚠ No subagent configured — using main model (may affect response speed)"

`condenser.service.ts` accepts an `endpoint` parameter obtained from a new utility `getCondenserEndpoint()` that returns either the subagent config or the main server config dynamically. No compression task is ever silently skipped due to endpoint configuration.

---

#### 4. JSON mode for subagent calls

**v4 spec says:** "JSON schema enforced where required, JSON mode"

**Reality:** `subagent.service.ts` makes plain `/v1/chat/completions` calls. No `response_format` is set anywhere in the codebase.

**Resolution:** All compression/reranker subagent calls add `response_format: { type: "json_object" }` to the request body. This is supported by llama.cpp server. A new helper `callSubagentForJson(prompt, systemPrompt, schema)` will be added to `subagent.service.ts` that wraps this pattern with automatic retry and schema validation via `zod` (already a project dependency).

---

#### 5. `@context-n` directive is ambiguous

**v4 spec says:** `@context-n` (manual injection)

**UI spec says:** Clicking a key "displays the @context-n directive in the prompt at cursor location"

**Problem:** `n` is never defined. Is it a numeric index? The UI shows key names, not numbers.

**Resolution:** The directive syntax is `@context:<key>` (e.g., `@context:api_auth_fix`). The UI clicking a Topic Map entry inserts `@context:<key>` at cursor position. The mutual exclusion with `@map` is preserved. The v4 spec's `@context-n` is renamed to `@context:<key>` everywhere.

---

#### 6. `current_delta` in EMA is undefined

**v4 spec says:** `EMA_new = α * current_delta + (1 - α) * EMA_prev` but never defines `current_delta`.

**Resolution:** `current_delta` = estimated token count of the **most recent assistant response** (not the user message). Rationale: assistant responses are the primary driver of context growth, and they're the thing we can observe post-stream. This value is available from `ChatMessageTimings` (token count) already passed back via `onTimings` callback.

---

#### 7. Topic Map storage location

**v4 spec says:** "Exists outside this system" for Tier A, but Tier B is unspecified.

**Resolution:** Topic Map (Tier B) is stored **per-conversation** in a new IndexedDB table `condenser_maps`. Rationale: TTL of 5 turns is conversation-scoped; the LRU cap of 10 entries is per conversation; the right panel should show entries relevant to the current chat. Requires a Dexie schema version bump (v2 → v3).

---

#### 8. Threshold percentages apply to `serverStore.contextSize`

**v4 spec says:** thresholds at 25/50/75%.

**Reality:** `serverStore.contextSize` is `props.default_generation_settings.n_ctx`. In external mode (non-llama.cpp backend), this may be `null`.

**Resolution:** When `contextSize` is `null`, fall back to a configurable absolute token limit (default: 4096). Show "Context size unavailable — using 4096 token fallback" in the settings slider.

---

#### 9. UI spec "four marker slider" vs three thresholds

**UI spec says:** "four marker slider"

**v4 spec says:** thresholds at 25/50/75% (three levels)

**Resolution:** The slider has **four markers**: 25% (Layer 0/write), 50% (Layer 1), 75% (Layer 2), 100% (hard limit / out-of-context). The fourth marker is fixed at 100% and non-adjustable. The first three are draggable within defined bounds.

---

#### 10. Right panel layout

**UI spec says:** A third right-side panel.

**Reality:** Current layout is two-panel only: `Sidebar.Root` (left) + `Sidebar.Inset` (main). There is no right panel infrastructure.

**Resolution:** Implement as a **collapsible right drawer**, toggled by a button in `ChatScreenHeader`. On desktop: fixed-width right panel (240px) sliding in, not a full sidebar (no shadcn Sidebar component). On mobile: an overlay sheet. This minimizes layout disruption. The panel is rendered inside `ChatScreen` (not in the root layout), so it's only visible on chat routes.

---

#### 11. EMA thresholds are user-adjustable

**v4 spec says:** Fixed base thresholds 25/50/75, ±10% adjustment.

**Resolution:** The base thresholds (25/50/75) are user-configurable via the slider (Layer 0 implementation). The EMA only adjusts the **effective** thresholds dynamically within the ±10% range. User-set base values are stored in settings; EMA state is in-memory (not persisted across sessions).

---

#### 12. `@new` soft limit (3 uses)

**v4 spec says:** "Soft limit: 3 uses"

**Resolution:** Counted per conversation session (in-memory, not persisted). At the third use, a toast warns "You've used @new 3 times this session — consider starting a new chat." The user can still proceed.

## Architecture Overview

```
ChatForm (input)
  ↓ directive parser ($lib/utils/condenser-directives.ts)
  ↓ @new → condenserStore.reset()
  ↓ @map → condenserStore.rerank(prompt)
  ↓ @context:<key> → condenserStore.injectEntry(key)
  ↓ normal message

chat.svelte.ts (send flow)
  ↓ condenserStore.processPreFlight(userMessage)  ← Layer 0
  ↓ condenserStore.checkAndCompress(convId)        ← Layer 1/2
  ↓ condenserStore.getInjectionContext()           ← injects refs
  ↓ ChatService.sendMessage(messages + injectedContext)

condenserStore ($lib/stores/condenser.svelte.ts)
  ├── EMA engine
  ├── threshold manager
  ├── compression pipeline (calls subagent.service.ts)
  └── Topic Map CRUD (calls DatabaseService.condenser*)

DatabaseService (version 3 schema)
  └── condenser_maps table: { convId, key, summary, rawJson, lastAccessed, turnCount }
```

### Directive Parser (`src/lib/utils/condenser-directives.ts`)

```typescript
export type ParsedDirective =
	| { type: 'new' }
	| { type: 'map' }
	| { type: 'context'; key: string }
	| { type: 'none' };

export function parseCondenserDirective(input: string): {
	directive: ParsedDirective;
	cleanedInput: string; // input with directive stripped
};
```

Parsing rules:

- `@new` anywhere in message → `{ type: 'new' }`, remove `@new` from sent text
- `@map` anywhere → `{ type: 'map' }`, remove from text
- `@context:<key>` (key = `[a-zA-Z0-9_-]+`) → `{ type: 'context', key }`, remove from text
- `@map` + `@context:*` in same message → error toast "Cannot use @map and @context together", block send
- Otherwise → `{ type: 'none' }`

### EMA Engine (`src/lib/utils/condenser-ema.ts`)

```typescript
export interface ThresholdConfig {
	baseL0: number; // % of context window
	baseL1: number;
	baseL2: number;
	alpha: number; // EMA smoothing
	maxAdjPct: number; // max adjustment = 10
}

export function updateEma(state: EmaState, delta: number): EmaState {
	return {
		value: state.alpha * delta + (1 - state.alpha) * state.value,
		alpha: state.alpha
	};
}

// Returns effective thresholds (base ± EMA adjustment, capped at ±10%)
export function computeEffectiveThresholds(
	config: ThresholdConfig,
	ema: EmaState,
	previousEma: number
): { l0: number; l1: number; l2: number };

// Returns the current context fill percentage
export function computeFillPct(estimatedUsedTokens: number, contextSize: number): number;

// Determines which layer (if any) should trigger
export function determineLayer(
	fillPct: number,
	thresholds: { l0: number; l1: number; l2: number },
	l1CooldownActive: boolean
): CondenserLayer;
```

**EMA current_delta definition:** The token count of the most recently received assistant message, extracted from `ChatMessageTimings.predicted_n` (already available via `onTimings` callback in chat.svelte.ts). Falls back to `estimateTokens(responseText)` when timings are unavailable.

**Forward-only adjustment:** `effective_l1 = base_l1 + clamp(ema_growth_factor * base_l1, 0, base_l1 * 0.10)`. Only increases (never decreases back toward base).

**cooldown logic:** After Layer 1 fires, store `tokensSinceLastL1 = 0`. Track tokens added each turn. Fire Layer 1 again only when `tokensSinceLastL1 / contextSize > condenserCooldownPct / 100`.

### WebWorker (`src/lib/workers/condenser-compression.worker.ts`)

```typescript
import pako from 'pako';

// Handles two message types from main thread:
// { type: 'compress', id: string, payload: string }  → gzip deflate + btoa
// { type: 'decompress', id: string, payload: string } → atob + inflate

self.onmessage = (e) => {
	const { type, id, payload } = e.data;
	try {
		if (type === 'compress') {
			const compressed = pako.deflate(payload);
			const b64 = btoa(String.fromCharCode(...compressed));
			self.postMessage({ id, result: b64 });
		} else if (type === 'decompress') {
			const bytes = Uint8Array.from(atob(payload), (c) => c.charCodeAt(0));
			const text = pako.inflate(bytes, { to: 'string' });
			self.postMessage({ id, result: text });
		}
	} catch (err) {
		self.postMessage({ id, error: String(err) });
	}
};
```

Worker is instantiated once as a module-level singleton in `condenser-snapshot.ts` and reused.

### Condenser Store — Compression Methods

Fill in `condenserStore` methods:

```typescript
// Called before sending a message (Layer 0)
async processPreFlight(convId: string, userMessage: string): Promise<string>

// Called after estimating token fill (Layer 1 or 2)
async checkAndCompress(convId: string, messages: DatabaseMessage[]): Promise<void>

// Returns injection string for @map or @context:<key>
async getInjectionForDirective(
  convId: string,
  directive: ParsedDirective
): Promise<string | null>

// Resets context for @new
async resetContext(convId: string): Promise<void>

// Updates EMA after each assistant response
updateEma(convId: string, assistantTokens: number): void

// Returns text to inject as system context (called in chat.svelte.ts)
getInjectionContext(convId: string): string | null
```

**`checkAndCompress` flow:**

1. Estimate total token usage from all messages in conversation
2. Compute fill pct vs contextSize (fallback to 4096 if null)
3. Determine layer via `determineLayer()`
4. If `layer1` and not in cooldown: create snapshot, run `compressTurnsPassive()`, save compressed turns, update cooldown
5. If `layer2`: re-run `recompressAggressive()` on existing compressed turns, write Topic Map entry
6. Update EMA state

**Topic Map write trigger:** After Layer 2 fires:

1. Extract the `goal` field from the **first** `CompressedTurn` in the newly re-compressed set — this becomes the auto-generated key (truncated to 40 chars, spaces replaced with `_`).
2. If that field is empty or `compression_failed`, fall back to `"segment_${timestamp}"`.
3. Write entry to `condenser_maps`. If a key collision occurs (same key already exists), append `_2`, `_3`, etc.
4. If entries ≥ 10, call `evictLruTopicMapEntry()` first.
5. Increment turn counts and evict TTL-expired entries (TTL = 5).
6. The key is **immediately editable** via the CondenserMapEditDialog — the edit icon in the panel allows renaming the key at any time.

## Cross-Layer Notes for the Implementing Agent

### What NOT to do

- Do not modify `ChatService.sendMessage()` directly — all condenser hooks go through `chat.svelte.ts`
- Do not add `response_format` to the main chat API calls — only to compression calls in `condenser.service.ts`
- Do not run pako synchronously on the main thread — always dispatch to the WebWorker in `condenser-compression.worker.ts`
- Do not add condenser state to `DatabaseMessage` or `DatabaseConversation` types — use the new `condenser_maps` table
- The `@new` directive inserts a separator `DatabaseMessage` (role `'separator'`) — do not clear visible messages

### Compression endpoint resolution

Use `getCondenserEndpoint()` from `condenser.service.ts` for all compression API calls. It returns either the subagent config or the main server config depending on what's available. Never hardcode endpoint logic inside the compression functions themselves.

### Pattern to follow for subagent calls

See `src/lib/services/subagent.service.ts` `summarizeText()` as the base pattern. New calls use `callSubagentForJson()` wrapper with zod schema validation.

### Pattern to follow for settings

See how `subagentEnabled`/`subagentEndpoint` are handled in `ChatSettings.svelte` (manual rendering, not the generic `ChatSettingsFields` component) — do the same for the Context tab.

### Pattern to follow for the right panel

See `+layout.svelte` for how `Sidebar.Root` works. For the condenser right panel, do NOT use `Sidebar.Root` — use a simple `div` with `transition:slide` (from svelte/transition) inside `ChatScreen.svelte`. The panel is local to the chat screen.

### Svelte 5 runes

All new stores must use `$state`, `$derived`, `$effect`. No Svelte 4 writable/readable stores. See `src/lib/stores/settings.svelte.ts` as the canonical pattern.

### Error handling contract

Per the v4 spec: **no cascade failures**. Every compression call that fails must:

1. Log to console with `[Condenser]` prefix
2. Set `compressionFailed: true` on the affected turn
3. Fall back to raw content
4. Show a subtle toast (non-blocking): "Compression failed for 1 turn — using original"
   Do not block the main chat send flow on compression errors.

## Resolved Decisions (Previously Open Questions)

All three open questions from the initial draft have been answered by the user:

1. **Layer 2 blocking dialog** → **Once per compaction event only.** After the user acknowledges ("Continue"), the dialog does not re-appear until the next new compaction event. See Layer 3 spec.

2. **`@new` and conversation history** → **Use a separator.** Visible messages stay intact. `condenserStore.compressedTurns[convId]` is cleared and a separator `DatabaseMessage` (role `'separator'`) is inserted rendering as `"── Context reset (@new) ──"`. The Topic Map active entries are reset.

3. **Topic Map auto-naming** → **First `goal` field from the first compressed turn**, truncated to 40 chars with spaces → `_`. User can rename it at any time via the edit/pen icon in the CondenserMapPanel.

4. **Topic Map auto-naming**: When Layer 2 fires and writes a new Topic Map entry, what key (name) should it get? The spec doesn't define this. Options:
   - Auto: timestamp string ("Session 2026-04-05 14:32")
   - Auto: first goal field from the most recent CompressedTurn
   - Prompt user for a name

   Recommendation: Use the **first goal field** as the auto-key (truncated to 40 chars). User can rename in the edit dialog.
