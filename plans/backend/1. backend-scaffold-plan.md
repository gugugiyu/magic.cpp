# Backend Scaffold Plan

**Status:** Ready to implement (clarifications resolved)
**Scope:** `packages/backend/` — Bun HTTP server that aggregates multiple llama-server/OpenAI-compatible upstreams into a single API surface for the frontend.

---

## What this backend does

The frontend talks to **one** server: this backend. It never knows about upstreams. The backend:

1. **Serves the frontend bundle** — static files from `packages/frontend/public/`
2. **Aggregates model lists** — queries all upstreams' model endpoints, merges them into a single `/v1/models` response
3. **Routes requests by model** — when a chat/load/unload request arrives, the backend identifies which upstream owns that model and forwards there
4. **Owns all auth** — API keys for upstreams live in backend config only; the frontend has no key awareness
5. **Provides `/cors-proxy`** — transparent HTTP proxy for cross-origin fetches (MCP resources, favicons)
6. **Heartbeats** — periodic health checks per upstream; marks degraded upstreams

Use case driving this: hybrid **offline main model** (local llama-server) + **online subagent** (external API), selectable from the frontend's model picker without the frontend knowing anything about where each model lives.

## Architecture

```
Browser (frontend)
      │
      ▼
┌─────────────────────────────────────────┐
│           packages/backend              │
│                                         │
│  GET  /v1/models    → merge all pools   │
│  POST /v1/chat/completions              │
│         → route by model → upstream N  │
│  GET  /props[?model=]                   │
│         → route to owning upstream      │
│  GET/POST /models, /models/load, …      │
│         → route to owning upstream      │
│  GET  /cors-proxy?url=…                 │
│         → transparent fetch proxy      │
│  GET  /health                           │
│         → pool status                  │
│  GET  /*, index.html                   │
│         → static frontend bundle       │
└─────────────────────────────────────────┘
         │                    │
         ▼                    ▼
  llama-server          OpenAI-compat API
  localhost:8080        api.openai.com
  (offline, main)       (online, subagent)
```

### Runtime
- **Bun** — `Bun.serve()`, no HTTP framework
- **TypeScript** — `tsconfig.json` targeting Bun

## Upstream config format

Two options. Picking the cleaner one for multi-upstream:

**`config.json`** for pool structure + **`.env`** for secrets.

- `config.json` is committed (alongside `config.example.json`); contains all non-secret config
- `.env` is gitignored; contains only API keys, referenced by name in `config.json`
- On startup, the backend merges both: resolves `"apiKey": "$OPENAI_KEY"` placeholders from env

```json
// config.json (committed)
{
  "port": 3000,
  "staticDir": "../frontend/public",
  "heartbeatInterval": 30,
  "upstreams": [
    {
      "id": "local-main",
      "label": "Local (main model)",
      "url": "http://localhost:8080",
      "type": "llamacpp",
      "apiKey": null,
      "enabled": false,
      "modelList": [
        "qwen/qwen-coder-next",
        "google/gemma3"
      ]
    },
    {
      "id": "openai-subagent",
      "label": "OpenAI (subagent)",
      "url": "https://api.openai.com",
      "type": "openai",
      "apiKey": "$OPENAI_KEY"
    }
  ]
}
```

```dotenv
# .env (gitignored)
OPENAI_KEY=sk-...
```

`type: "llamacpp"` — supports `/props`, `/models/load`, `/models/unload`
`type: "openai"` — supports `/v1/models`, `/v1/chat/completions` only; `/props` is synthesized

## Key implementation details

### Model routing
On startup (and after each heartbeat), `model-pool.ts` fetches `/v1/models` from each upstream and builds a `Map<modelId, upstreamId>`. Requests with `?model=` or `"model"` in the JSON body are looked up in this map to select the upstream.

### Streaming proxy
`/v1/chat/completions` uses SSE. The proxy must:
- Forward the request body as-is (inject auth header)
- Return `Transfer-Encoding: chunked` or stream directly via `ReadableStream`
- Not buffer the response

### `/props` synthesis for OpenAI upstreams
OpenAI-type upstreams have no `/props`. The backend returns a synthetic props object with sensible defaults (same shape as `PropsService.DEFAULT_PROPS` in the frontend) so the frontend's role detection still works. Role will be `MODEL` for synthesized props.

### `/cors-proxy` header unwrapping
Frontend sends headers prefixed `x-proxy-header-<name>`. The handler strips the prefix and forwards as plain headers to the target URL.

### Static serving + SPA fallback
All unmatched GET routes return `index.html` so SvelteKit's client-side routing works.

## Frontend changes required (separate branch)

The frontend currently owns API key + endpoint config in `apiConfigStore`. With the backend as the abstraction layer, this needs to be removed:

- Remove `VITE_API_BASE_URL` / `VITE_API_KEY` env vars
- Remove `ApiMode.OPENAI` / `isExternalMode()` code paths — there is no "external mode" anymore; the backend IS the server
- Remove `apiConfigStore` API key / endpoint settings UI
- Remove `apiKey` from `settingsStore` config
- `CORS_PROXY_ENDPOINT` stays — the backend implements it

> This is non-trivial frontend work. Treat it as a separate branch/phase after the backend MVP is running.

## Docker Compose (docker/)

```yaml
# docker/compose.yml
services:
  backend:
    build:
      context: ../packages/backend
      dockerfile: Dockerfile
    ports:
      - "3000:3000"
    volumes:
      - ./config.json:/app/config.json:ro    # mount pool config
    restart: unless-stopped

  llama-server:
    image: ghcr.io/ggerganov/llama.cpp:server
    ports:
      - "8080:8080"
    volumes:
      - ${MODELS_DIR}:/models
    command: ["-m", "/models/${MAIN_MODEL}", "--port", "8080", "--host", "0.0.0.0"]
    restart: unless-stopped
```

Online upstreams (OpenAI, etc.) are just URLs in `config.json` — no containers needed.
