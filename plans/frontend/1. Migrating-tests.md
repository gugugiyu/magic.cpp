# Test Migration Plan for Frontend Decoupling

**Goal:** Adapt existing tests to work with the new configurable server endpoint (from `0. Decoupling.md`).

**Background:** The decoupling implementation has already been partially completed - `serverEndpointStore` exists and `apiFetch` already uses it. This plan focuses on ensuring tests work correctly with this new architecture.

---

## Current Test Structure

| Type | Location | Count | Makes Network Calls |
|------|----------|-------|---------------------|
| Unit | `tests/unit/` | 8 files | No (pure functions) |
| Client | `tests/client/` | 1 file | Possibly (Svelte components) |
| E2E | `tests/e2e/` | 1 file | Yes (requires server) |
| Stories | `tests/stories/` | 5 stories | No (isolated) |

---

## Findings

### 1. Tests Already Compatible

Most unit tests (`uri-template.test.ts`, `reasoning-context.test.ts`, `model-names.test.ts`, `latex-protection.test.ts`, `clipboard.test.ts`, `agentic-strip.test.ts`, `agentic-sections.test.ts`) test pure functions and don't make network calls - no changes needed.

### 2. Service Tests Using Static Methods

The `model-id-parser.test.ts` imports `ModelsService` but only uses `parseModelId()` which is a pure function - no changes needed.

### 3. E2E Tests

Current E2E test (`demo.test.ts`) navigates to `/` - requires a running server. With configurable endpoints, users can test against different backends.

### 4. Storybook Stories

Stories render components in isolation - they don't make real API calls. They may need mocks for server-dependent components.

---

## Implementation Plan

### Phase 1: Test the Server Endpoint Store

**New test file:** `tests/unit/server-endpoint.test.ts`

```typescript
// Test cases needed:
- Initial default state (localhost:8080, isDefault: true)
- setBaseUrl() updates config and persists to localStorage
- setDefault() resets to localhost:8080
- getBaseUrl() returns correct URL based on isDefault state
- Loading from localStorage on initialization
- Validation of URL format
```

---

### Phase 2: Test the API Fetch Utility

**New test file:** `tests/unit/api-fetch.test.ts`

```typescript
// Test cases needed:
- Uses relative path when isDefault() is true
- Prepends custom base URL when isDefault() is false
- Passes through absolute URLs unchanged
- Handles query parameters correctly
- Error handling for failed requests
```

**Note:** These tests require mocking `fetch` and the `serverEndpointStore`.

---

### Phase 3: Mock Server Endpoint in Existing Tests

For tests that might indirectly depend on server configuration:

1. **vitest-setup.ts** - Add global mock for `serverEndpointStore`:
   ```typescript
   vi.mock('$lib/stores/server-endpoint.svelte', () => ({
     serverEndpointStore: {
       isDefault: () => true,
       getBaseUrl: () => '',
       getConfig: () => ({ baseUrl: 'http://localhost:8080', isDefault: true })
     }
   }))
   ```

2. **Storybook vitest setup** - Ensure stories work with mocked store

---

### Phase 4: E2E Test Updates

1. Add test configuration for custom endpoint:
   ```typescript
   // playwright.config.ts
   const customServerUrl = process.env.TEST_SERVER_URL || 'http://localhost:8080';
   ```

2. Add E2E tests for Connection tab:
   - `tests/e2e/connection-settings.test.ts`
   - Test endpoint input field
   - Test reset to default
   - Test model refresh after endpoint change

---

### Phase 5: CI/CD Integration

Update test commands to optionally accept a custom server URL:

```bash
# Test against default (localhost:8080)
npm run test

# Test against custom endpoint
TEST_SERVER_URL=http://localhost:3000 npm run test:e2e
```
