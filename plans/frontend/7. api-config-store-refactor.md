# Frontend: apiConfigStore Refactor Plan

**Status:** Pending — blocked on backend MVP (Phase 1 + 2) being functional
**Branch:** to be created (do NOT merge to master before backend is live)

---

## Why

With the backend acting as the abstraction layer, the frontend no longer needs to know about:
- External API endpoints (no more `VITE_API_BASE_URL`)
- API keys (no more `VITE_API_KEY`, no key in settings UI)
- The distinction between `llamacpp` and `openai` mode (`ApiMode` enum, `isExternalMode()`)

The frontend's only job is: pick a model, send a request to **this server** (the backend), display the response.

---

## What gets removed / changed

### Delete entirely
- `src/lib/enums/api-mode.ts` — `ApiMode` enum
- `src/lib/stores/api-config.svelte.ts` — entire store
- `src/lib/utils/api-headers.ts` — `getAuthHeaders()`, `getJsonHeaders()` (auth header injection moves to backend)
- `VITE_API_BASE_URL` and `VITE_API_KEY` from `.env` / `vite.config.ts` docs

### Simplify
- `src/lib/stores/settings.svelte.ts` — remove `apiEndpoint`, `apiKey`, `apiMode` from config schema and defaults
- `src/lib/constants/settings-keys.ts` — remove `API_ENDPOINT`, `API_KEY`, `API_MODE` keys
- `src/lib/constants/settings-config.ts` — remove those default values + their descriptions
- `src/lib/services/props.service.ts` — remove the `isExternalMode()` branch; always call `/props` directly
- `src/lib/stores/server.svelte.ts` — remove `isExternalMode()` / `apiConfigStore.setConnecting()` calls; simplify `fetch()` to always use `PropsService.fetch()`

### UI
- Remove the API key / endpoint / mode fields from the settings dialog (wherever they currently appear in `dialogs/`)
- `connectionStatus` display (if any) can be simplified — there's only one server now

### Vite proxy (dev only)
`vite.config.ts` proxy still points to `:8080` for development without the backend. After this refactor, dev proxy should point to the backend (`:3000`) instead. Update:
```ts
server: {
  proxy: {
    '/v1': 'http://localhost:3000',
    '/props': 'http://localhost:3000',
    '/models': 'http://localhost:3000',
    '/cors-proxy': 'http://localhost:3000'
  }
}
```

---

## What stays

- `CORS_PROXY_ENDPOINT` constant and `cors-proxy.ts` util — the backend still implements this endpoint
- `getJsonHeaders()` — keep but without the auth part (just `Content-Type: application/json`)
- All store/service logic unrelated to auth/endpoint config

---

## Dependency order

1. Backend Phase 1 (static serve + single-upstream proxy) must be running
2. Backend Phase 2 (multi-upstream pool) must be running
3. Then do this refactor — verify frontend works end-to-end against backend before deleting code
4. Remove dead code incrementally, run `bun run check` + `bun run test:unit` after each removal

---

## Risk

`apiConfigStore.connectionStatus` is used in some UI component to show a connection indicator. Check all usages before deleting the store — the indicator may need to be re-wired to the backend's `/health` endpoint response instead.

```bash
# Find all consumers before touching anything
grep -r "apiConfigStore\|isExternalMode\|ApiMode\|apiMode\|apiEndpoint\|apiKey\|VITE_API" \
  packages/frontend/src/ --include="*.ts" --include="*.svelte" -l
```
