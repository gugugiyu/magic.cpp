## Memory injection — three tiers

When memory is enabled and the system message is being built (in `chat.service.ts`):

1. **User profile (always included)** — non-null fields from settings injected verbatim:
   `Name: Alice | Email: alice@example.com | …`
2. **10 most recent entries** — heuristic: user most likely wants recently-saved facts.
3. **Up to 4 contextually-relevant entries** — case-insensitive word-by-word match against the last user message against all memory keys. Capped at 4 to avoid bloat. This is fragile by design (acknowledged).
4. **Model-proactive fetching** — system prompt instructs/encourages the model to call `fetch_memories` if the injected entries are insufficient.

Combined context is appended to the system message in `chat.service.ts` (same pattern as language pinner — mutates the in-memory `normalizedMessages` only, never the DB).

## Architecture overview

```
IndexedDB (Dexie v2)
  └→ memories table  (id, key, value, timestamp)
       └→ memory.service.ts   (CRUD)
            └→ memory.svelte.ts  (reactive store)

chat.svelte.ts
  ├→ reads memory store before streamChatCompletion()
  ├→ builds memoryContext string
  └→ passes via SettingsChatServiceOptions.memoryContext

chat.service.ts
  ├→ substituteVariables()    ← personalization.ts
  ├→ injects memory context into system message
  └→ injects built-in tool definitions into requestBody.tools

agentic.svelte.ts
  ├→ getConfig(): enabled = mcpHasServers || anyBuiltinToolEnabled
  └→ tool execution loop: check built-in names first, fall through to MCP

ChatMessageAssistant.svelte
  ├→ [TL;DR block]       ← above message content
  ├→ [existing content]
  └→ [Links section]     ← below content, above model info
```

### 1. NEW `src/lib/types/memory.d.ts`

```typescript
export interface MemoryEntry {
	id: string; // nanoid
	key: string; // user-visible label (e.g. "favorite color")
	value: string; // stored value (e.g. "blue")
	timestamp: number; // Date.now() at save time
}
```

### 2. NEW `src/lib/services/memory.service.ts`

Thin wrapper around Dexie `memories` table:

```typescript
export class MemoryService {
	async getAll(): Promise<MemoryEntry[]>;
	async getRecent(limit: number): Promise<MemoryEntry[]>; // ordered by timestamp DESC
	async search(query: string, limit: number): Promise<MemoryEntry[]>; // word-by-word match
	async save(key: string, value: string): Promise<void>; // upsert by key (case-insensitive)
	async delete(id: string): Promise<void>;
	async clear(): Promise<void>;
	async count(): Promise<number>;
}
```

`search()`: split query into words, filter entries where `entry.key` or `entry.value` contains any word (case-insensitive), return top `limit` by timestamp DESC.

`save()` upserts: if a memory with the same key already exists (case-insensitive), update its value + timestamp. If at capacity (100), delete the oldest entry before inserting.

### 13. MODIFY `src/lib/services/chat.service.ts`

After the language pinner block (and before `requestBody` is built), insert two new steps:

**Step A — Variable substitution** (replaces `{{USERNAME}}` etc. in system message):

```typescript
if (anyPersonalizationFieldSet) {
	const systemMsg = normalizedMessages.find((m) => m.role === MessageRole.SYSTEM);
	if (systemMsg && typeof systemMsg.content === 'string') {
		systemMsg.content = substituteVariables(systemMsg.content, profile);
	}
}
```

**Step B — Memory context injection** (appends memory block to system message):

```typescript
if (options.memoryContext) {
	const systemMsg = normalizedMessages.find((m) => m.role === MessageRole.SYSTEM);
	if (systemMsg && typeof systemMsg.content === 'string') {
		systemMsg.content += '\n\n' + options.memoryContext;
	} else if (!systemMsg) {
		normalizedMessages.unshift({ role: MessageRole.SYSTEM, content: options.memoryContext });
	}
}
```

**Step C — Built-in tool injection** (merge into `requestBody.tools`):

```typescript
if (options.builtinTools?.length) {
	requestBody.tools = [...(requestBody.tools ?? []), ...options.builtinTools];
}
```

## Variable substitution tokens

| Token          | Settings key                  | Profile field |
| -------------- | ----------------------------- | ------------- |
| `{{USERNAME}}` | `personalizationUserName`     | name          |
| `{{EMAIL}}`    | `personalizationUserEmail`    | email         |
| `{{GENDER}}`   | `personalizationUserGender`   | gender        |
| `{{BIO}}`      | `personalizationUserBio`      | bio           |
| `{{BIRTHDAY}}` | `personalizationUserBirthday` | birthday      |

Substitution runs case-insensitively on the system message content only, in `chat.service.ts`, before the request is sent. The substituted string is never persisted.
