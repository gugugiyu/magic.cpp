# Frontend Decoupling Plan

**Goal:** Decouple the frontend from llama-server, enabling it to work with the new backend aggregation layer (or any OpenAI-compatible server).

**Scope:** `packages/frontend/` — Allow the UI to be configured to point at a different server URL instead of being hardcoded to llama-server.

---

## Current State

### Hardcoded Endpoints

The frontend is **tightly coupled** to a local llama-server instance. The only place the server URL is configured is:

**`vite.config.ts` (lines 166-169)** — Development proxy only:
```typescript
server: {
  proxy: {
    '/v1': 'http://localhost:8080',
    '/props': 'http://localhost:8080',
    '/models': 'http://localhost:8080',
    '/cors-proxy': 'http://localhost:8080'
  }
}
```

### No Runtime Flexibility

- All API calls use **relative paths** (`./v1/chat/completions`, `./props`, etc.)
- The `$app/paths` base is used, but this resolves to relative paths in static builds
- **No environment variable** for API base URL at runtime
- **No input field** for users to configure a custom server endpoint
- The Vite proxy only works in development; production builds have no endpoint configuration

### Service Layer

| Service | File | Current Behavior |
|---------|------|------------------|
| `ChatService` | `chat.service.ts` | Relative `./v1/chat/completions` |
| `ModelsService` | `models.service.ts` | Relative `./v1/models` |
| `PropsService` | `props.service.ts` | Relative `./props` |

All services rely on same-origin requests via Vite proxy (dev) or same-origin in production.

---

## Strategy

1. **Backend as proxy** — External API calls go through the backend (not direct browser→external API), avoiding CORS issues entirely
2. **Single configuration** — The frontend only needs to know the server base URL
3. **localStorage persistence** — Server endpoint config is saved to localStorage and persists across sessions

This aligns with the backend plan (`plans/backend/1. backend-scaffold-plan.md`) which will:
1. Serve the frontend bundle
2. Aggregate llama-server + external OpenAI APIs
3. Handle all routing, auth, and CORS

## Implementation Phases

### Phase 1: Server Endpoint Store

**New store: `server-endpoint.svelte.ts`**

```typescript
interface ServerEndpointConfig {
  baseUrl: string;     // e.g., "http://localhost:3000" or "http://localhost:8080"
  isDefault: boolean;  // true = use hardcoded default (localhost:8080)
}

// State:
// #config = $state<ServerEndpointConfig>({ baseUrl: '', isDefault: true })

// Methods:
// getBaseUrl()        — returns configured or default base URL
// setBaseUrl(url)     — configure custom server URL
// setDefault()        — reset to hardcoded default (localhost:8080)
// isDefault()         — returns true if using default
```

**Storage:** LocalStorage key `LlamaCppWebui.serverEndpoint`

**Default:** `{ baseUrl: 'http://localhost:8080', isDefault: true }`

**Exported reactive accessors:**
```typescript
export const serverEndpointConfig = () => serverEndpointStore.config;
export const isUsingDefaultEndpoint = () => serverEndpointStore.isDefault();
export const getServerBaseUrl = () => serverEndpointStore.getBaseUrl();
```

---

### Phase 2: API Fetch Utility Refactor

**`api-fetch.ts` changes:**

Current URL resolution:
```typescript
const base = '';  // from $app/paths, which is relative
const url = path.startsWith('http') ? path : `${base}${path}`;
```

New URL resolution:
```typescript
const base = serverEndpointStore.isDefault()
  ? ''  // relative path for same-origin
  : serverEndpointStore.getBaseUrl();  // absolute URL
const url = path.startsWith('http') ? path : `${base}${path}`;
```

**Key consideration:** When `isDefault()` is true, we still use relative paths (same behavior as now). When `isDefault()` is false, we prepend the configured base URL.

### Phase 5: Server Store Integration

**`server.svelte.ts` changes:**

The `serverStore.fetch()` method needs to use the configured endpoint:

```typescript
async fetch(): Promise<void> {
  this.loading = true;
  this.error = null;
  try {
    const baseUrl = serverEndpointStore.getBaseUrl();
    const props = await PropsService.fetch(baseUrl);
    // ... rest unchanged
  } catch (e) {
    // ... error handling
  }
}
```

**Note:** When `isDefault()` is true, `getBaseUrl()` returns empty string, so relative paths are used (same-origin). When `isDefault()` is false, it returns the configured absolute URL.

## Relationship to Backend Plan

This plan enables the frontend to work with the upcoming backend:

| Phase | Frontend | Backend |
|-------|----------|---------|
| **Current** | Hardcoded to `:8080` (llama-server) | Not started |
| **This plan Phase 1-5** | Configurable endpoint | Not started |
| **Backend Phase 1-2** | Works with backend | Serves frontend, proxies to llama-server |
| **Backend Phase 4** | Endpoint config removed | Backend handles everything |

**Final state after backend Phase 4:** Frontend default becomes `localhost:3000` (backend), with option to point directly at `localhost:8080` (llama-server) for development.
