## 1. Executive Summary

The `call_subagent` built-in tool currently:
- Uses `stream: false`, blocking the main agentic loop until the full JSON response arrives (bad UX).
- Discards the entire subagent conversation history after execution (no observability).
- Stores progress state in dead frontend-only reactive maps that are keyed by `conversationId`, causing collisions when multiple subagents run in parallel.

This plan addresses both issues by:
1. **Persisting subagent messages** in the existing SQLite `messages` table via a new `subagent_session_id` column.
2. **Refactoring the subagent executor** to use the same streaming primitives as the main chat (`ChatService.sendMessage` with `stream: true`).
3. **Building a dedicated subagent dialog** that renders historical subagent traces (post-hoc) and optionally subscribes to live updates (real-time).
4. **Extracting a shared Agentic Loop Harness** to eliminate duplication between the main agentic loop and the subagent loop.

---

## 2. Guiding Principles

| Principle | Rationale |
|---|---|
| **Reuse the `messages` table** | Avoids a new table, keeps branching/children logic, and reuses existing API patterns. |
| **`toolCallId` as session key** | Already unique per invocation, stable across retries, and present in the parent tool call row. |
| **Filter, don't fork** | Main chat queries must explicitly exclude subagent rows so existing conversation loading logic is unaffected. |
| **Shared harness** | Main chat (`chat.svelte.ts` → `agenticStore`) and subagent (`agentic-builtin-tools.service.ts`) both perform multi-turn streaming + tool execution. Extracting this reduces bug surface. |
| **Immutable after completion** | Subagent dialog is read-only; no input form. History is frozen once the subagent returns its final result to the main model. |

## 5. Shared Agentic Loop Harness (Phase 2)

### 5.1 Problem Statement

Both the **main chat** and the **subagent** execute the same high-level loop:
1. Stream an assistant message.
2. Detect tool calls.
3. Execute tools (parallel).
4. Persist tool results.
5. Append to message history.
6. Repeat until max turns or no tool calls.

### 5.2 Implementation: `AgenticLoopHarness`

**File:** `packages/frontend/src/lib/services/agentic/agentic-loop-harness.service.ts` ✅ **CREATED**

**Responsibilities:**
- Run one turn of streaming LLM + tool execution.
- **NOT** responsible for: conversation-level state, MCP connection lifecycle, or UI stores.

**Interface (as implemented):**

```ts
export interface AgenticTurnCallbacks {
	onChunk?: (chunk: string) => void;
	onReasoningChunk?: (chunk: string) => void;
	onToolCallChunk?: (serializedToolCalls: string) => void;
	onModel?: (model: string) => void;
	onTimings?: (timings?: ChatMessageTimings, promptProgress?: ChatMessagePromptProgress) => void;
}

export interface AgenticTurnResult {
	content: string;
	reasoningContent?: string;
	toolCalls?: ApiChatCompletionToolCall[];
	timings?: ChatMessageTimings;
}

export interface ToolCallOutcome {
	toolCall: ApiChatCompletionToolCall;
	result: string;
	success: boolean;
	durationMs: number;
	extras?: DatabaseMessageExtra[];
	/** Content parts for vision model session history (includes image attachments). */
	contentParts?: ApiChatMessageContentPart[];
}

export interface ExecuteBuiltinToolFn {
	(
		name: string,
		args: string,
		conversationId: string,
		messageId: string,
		toolCallId?: string
	): Promise<{ content: string; extras?: DatabaseMessageExtra[] }>;
}

export interface ExecuteToolsOptions {
	registry: AgenticToolRegistry;
	conversationId: string;
	messageId: string;
	allTools?: OpenAIToolDefinition[];
	signal?: AbortSignal;
	effectiveModel?: string;
	mcpSummarizeOutputs: boolean;
	mcpSummarizeLineThreshold: number;
	mcpSummarizeHardCap: number;
	mcpSummarizeAllTools: boolean;
	executeBuiltinTool: ExecuteBuiltinToolFn;
}

export class AgenticLoopHarness {
	static async runSingleTurn(
		messages: ApiChatMessageData[],
		options: { model?: string; endpoint?: string; apiKey?: string; tools?: OpenAIToolDefinition[]; signal?: AbortSignal; /* ...sampling params */ },
		callbacks: AgenticTurnCallbacks
	): Promise<AgenticTurnResult>;

	static async executeTools(
		toolCalls: ApiChatCompletionToolCall[],
		options: ExecuteToolsOptions
	): Promise<ToolCallOutcome[]>;
}
```

**Key design decisions:**
- `runSingleTurn` delegates to `ChatService.sendMessage` with `stream: true`. It now supports `endpoint` and `apiKey` overrides (added to `SettingsChatServiceOptions`) so the subagent can hit a different upstream.
- `executeTools` accepts an `executeBuiltinTool` **callback** so callers can inject their own builtin dispatch. This avoids a circular dependency: the harness cannot import `AgenticBuiltinToolExecutor` because the executor will import the harness for the subagent loop.
- `ToolCallOutcome` includes `contentParts` for vision-model session history. The caller decides whether to use `result` (string) or `contentParts` (array) when appending to message history.

**Migration Path (completed):**
1. ✅ Implemented `AgenticLoopHarness.runSingleTurn` by extracting core streaming logic from `agenticStore.executeAgenticLoop`.
2. ✅ Implemented `AgenticLoopHarness.executeTools` by extracting tool execution + summarize/attachment logic.
3. ✅ Updated `agenticStore.executeAgenticLoop` to delegate each turn to `AgenticLoopHarness.runSingleTurn` and each tool phase to `AgenticLoopHarness.executeTools`.
4. ✅ Exported from `packages/frontend/src/lib/services/index.ts`.

> **Note:** `agentic.svelte.ts` now references `ChatService.convertDbMessageToApiChatMessageData` directly (not via the harness). The harness does not own message-format conversion; that remains on `ChatService`.

## 6. Subagent Streaming & Persistence (Phase 3)

### 6.1 Streaming the Subagent

Refactor `AgenticBuiltinToolExecutor.executeCallSubagent()` to:
1. Use `AgenticLoopHarness.runSingleTurn` instead of raw `fetch` + `response.json()`.
2. Run a `while` loop (max 10 turns) using the **shared harness** from Phase 2.
3. Stream tokens into `subagentSessionStore` (and local accumulator).
4. When `onToolCallChunk` fires, collect tool calls.
5. Execute collected tools via `AgenticLoopHarness.executeTools`.
6. Persist assistant + tool messages to DB with `subagentSessionId = toolCallId`.
7. Append assistant + tool messages to `loopMessages`.
8. Continue until no tool calls or max turns.

> **Status:** ✅ **COMPLETE.** Refactored to use `AgenticLoopHarness.runSingleTurn` + `executeTools`, with streaming callbacks updating `subagentSessionStore` and DB persistence at turn boundaries via `DatabaseService.createMessageBranch` with `subagentSessionId`.

### 6.2 Persistence Strategy

Each subagent turn writes to the DB **at turn boundaries** (not per-token) so the dialog can render history in real-time or post-hoc without overwhelming the backend with HTTP requests.

**Message creation flow inside `executeCallSubagent`:**

```ts
// Before first turn:
const sessionId = toolCallId; // the call_subagent toolCallId

// System message
const systemMsg = await DatabaseService.createMessageBranch(
	{ convId, type: 'text', role: 'system', content: system || SUBAGENT_DEFAULT_PROMPT, timestamp: Date.now(), subagentSessionId: sessionId },
	null
);

// For each turn:
// 1. Stream assistant turn via AgenticLoopHarness.runSingleTurn
const turnResult = await AgenticLoopHarness.runSingleTurn(loopMessages, { ... }, callbacks);

// 2. Persist assistant message
const assistantMsg = await DatabaseService.createMessageBranch(
	{ convId, type: 'text', role: 'assistant', content: turnResult.content, reasoningContent: turnResult.reasoningContent, toolCalls: JSON.stringify(turnResult.toolCalls), model: subagentModelId, timestamp: Date.now(), subagentSessionId: sessionId },
	lastMessageId
);
lastMessageId = assistantMsg.id;

// 3. If tool calls exist, execute and persist results
const outcomes = await AgenticLoopHarness.executeTools(turnResult.toolCalls, { ... });
for (const outcome of outcomes) {
	const toolMsg = await DatabaseService.createMessageBranch(
		{ convId, type: 'text', role: 'tool', content: outcome.result, toolCallId: outcome.toolCall.id, extra: outcome.extras, timestamp: Date.now(), subagentSessionId: sessionId },
		lastMessageId
	);
	lastMessageId = toolMsg.id;
}
```

**Critical discovery:** `DatabaseService.createMessageBranch` was **not** passing `subagentSessionId` through to the API layer. This was fixed in the current session by adding the field to the `messagesAPI.createMessageBranch` call inside `database.service.ts`.

### 6.3 Abort Signal Propagation

- ✅ `AgenticLoopHarness.runSingleTurn` forwards `signal` to `ChatService.sendMessage`.
- ✅ `AgenticLoopHarness.executeTools` forwards `signal` to `mcpStore.executeTool` and builtin tool execution.
- The subagent should use `createLinkedController(signal)` so that aborting the main chat also aborts the subagent stream.

### 6.4 Token Counting During Streaming

- Token counts are not available per-chunk during streaming.
- `subagentSessionStore` tracks `totalTokens`, `promptTokens`, `completionTokens` and exposes them reactively.
- When the upstream reports `usage` (non-streaming path or stream end), update the store. For streaming, show a "Streaming…" indicator and backfill counts when available.

### 7.1 New Store: `subagentSessionStore`

**File:** `packages/frontend/src/lib/stores/subagent-session.svelte.ts` ✅ **CREATED**

**Responsibilities:**
- Track which subagent sessions exist, keyed by `sessionId` (= `toolCallId`).
- Provide reactive state for the **live** subagent (streaming content, steps, token counts).
- Store is global; components look up sessions by `toolCallId`.

**Interface (as implemented):**

```ts
export interface SubagentStep {
	toolName: string;
	status: 'calling' | 'done';
}

export interface SubagentSession {
	sessionId: string;
	conversationId: string;
	isRunning: boolean;
	isComplete: boolean;
	currentTurn: number;
	modelName: string;
	content: string;
	reasoningContent: string;
	steps: SubagentStep[];
	totalTokens: number;
	promptTokens: number;
	completionTokens: number;
	toolCallsCount: number;
	originSkill?: string;
	error: string | null;
}

class SubagentSessionStore {
	startSession(params: { sessionId; conversationId; modelName; originSkill? }): void;
	appendContent(sessionId: string, chunk: string): void;
	appendReasoning(sessionId: string, chunk: string): void;
	setModel(sessionId: string, modelName: string): void;
	setTurn(sessionId: string, turn: number): void;
	addStep(sessionId: string, toolName: string, status?: 'calling' | 'done'): void;
	updateStep(sessionId: string, toolName: string, status: 'calling' | 'done'): void;
	incrementToolCallsCount(sessionId: string): void;
	addUsage(sessionId: string, prompt: number, completion: number, total: number): void;
	setError(sessionId: string, error: string): void;
	completeSession(sessionId: string): void;
	clearSession(sessionId: string): void;
	getSession(sessionId: string): SubagentSession | undefined;
	isRunning(sessionId: string): boolean;
	isComplete(sessionId: string): boolean;
}

export const subagentSessionStore = new SubagentSessionStore();
```

## 13. Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| **Schema migration failure** on existing user DBs | High | Use Drizzle's `migrate()` which handles incremental migrations. Test against a copy of a production-sized DB. |
| **Shared harness refactor breaks main agentic flow** | High | Phase 2 must be completed and fully tested in isolation before Phase 3 begins. No other features touch `agenticStore` during this window. |
| **DB write frequency during streaming** | Medium | Debounce writes to SQLite (frontend→backend HTTP calls are expensive). Keep live UI reactive via local Svelte state; persist at turn boundaries. |
| **Nested abort signals** | Medium | Ensure `createLinkedController(signal)` is used so aborting the main chat also aborts the subagent stream. |
| **Dialog performance with long traces** | Low | Virtualize the message list if a subagent exceeds ~50 messages (unlikely with 10-turn max). |
| **Token count unavailable during streaming** | Low | Show "Streaming…" placeholder; backfill actual counts from `usage` when the stream completes. |

### 16.3 Critical Technical Changes from Original Plan

| Original Assumption | What Changed | Rationale |
|---|---|---|
| `AgenticLoopHarness` would own `convertDbMessageToApiChatMessageData` | Still lives on `ChatService` | Message format conversion is not part of the turn harness; it's a data-layer concern. The harness operates on `ApiChatMessageData[]`. |
| `executeCallSubagent` would call `ChatService.sendMessage` directly | Calls `AgenticLoopHarness.runSingleTurn` instead | Cleaner separation; the harness encapsulates streaming accumulation and callback wiring. |
| `AgenticLoopHarness.executeTools` would import `AgenticBuiltinToolExecutor` directly | Accepts `executeBuiltinTool` callback parameter | Avoids circular dependency: the executor imports the harness, so the harness cannot import the executor. |
| `DatabaseService.createMessageBranch` already supported `subagentSessionId` | **It did not.** The field was missing from the passthrough. | Discovered during implementation of Phase 3. Fixed in a prior session. |
| `_lastReadSkill` in `agenticStore` would associate skills with subagents | **Now dead code.** `read_skill` moved to `AgenticBuiltinToolExecutor`, so the store never sets it. | If `originSkill` association is still desired, it must be passed explicitly to `executeCallSubagent` or tracked in the store before delegation. |
| Non-streaming path in `ChatService.sendMessage` returns a `Response` | `apiPost` returns parsed JSON, not `Response`. The non-streaming path casts the result via `as unknown as Response` to preserve the existing interface. | This is a legacy quirk of the `ChatService` API. The subagent only uses the streaming path, so this does not affect Phase 3. |
| Prop-drilling dialog state through component tree | Used `subagentDialogStore` instead | Avoids threading callbacks through 5 layers (ChatScreen → ChatMessages → ChatMessage → ChatMessageAssistant → ChatMessageAgenticContent). |

### 16.4 Bad Assumptions Discovered

1. **Backend subagent support was "to be built" — it was already built.**
   - The migration, schema, queries, handlers, and routes for `subagent_session_id` were already committed.
   - This accelerated Phase 1 significantly but also meant the plan's backend sections were partially redundant.

2. **`DatabaseService.createMessageBranch` passthrough completeness.**
   - Assumed all `DatabaseMessage` fields were forwarded to the API. `subagentSessionId` was silently dropped, which would have caused subagent messages to appear in the main chat.

3. **`AgenticBuiltinToolExecutor` receives `toolCallId` for all tools.**
   - The `toolCallId` parameter exists on `executeBuiltinTool`, but `executeCallSubagent` was **not** receiving it. Without this, there's no stable `subagent_session_id` for DB persistence.

4. **The main agentic loop and subagent loop can share the exact same `executeBuiltinTool` callback.**
   - They *can*, but the subagent must inject `allTools` and `signal` via closure because the `ExecuteBuiltinToolFn` interface has a narrower signature than `AgenticBuiltinToolExecutor.executeBuiltinTool`.

5. **`agentic.svelte.ts` refactor introduced type errors.**
   - `stream: true` and `parallel_tool_calls: true` were passed to `AgenticLoopHarness.runSingleTurn`, which does not accept them (it sets them internally). This caused a TypeScript error.
   - `toolCall.id` was used directly as `tool_call_id` even though it can be `undefined`.
   - `loading-context.svelte.ts` still referenced `agenticStore.subagentProgress`, which had been removed during the refactor.
